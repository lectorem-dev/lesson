# Техническое задание MVP платформы онлайн-обучения

## 1. Цель

Разработать веб-платформу для прохождения одного обучающего курса с последовательным открытием уроков, тестированием, учетом прогресса и скачиванием PDF-сертификата после успешного завершения курса.

MVP ориентирован на демонстрацию базового учебного сценария:

- администратор создает и управляет пользователями;
- студент проходит уроки строго по порядку;
- после каждого урока студент проходит тест;
- после завершения всех уроков студент может скачать сертификат.

## 2. Технологический стек

### Backend

- Python.
- FastAPI.
- SQLAlchemy 2.x.
- Alembic.
- PostgreSQL.
- JWT Authentication.
- Pydantic.
- ReportLab для генерации PDF-сертификата.

### Frontend

- React.
- TypeScript.
- Vite.
- React Router.
- Axios.
- react-markdown.

### Инфраструктура

- Docker.
- Docker Compose.
- nginx для Docker-сборки frontend.
- Swagger/OpenAPI, встроенный в FastAPI.

## 3. Состав проекта

- `backend/` - API, бизнес-логика, модели данных, миграции, импорт контента.
- `frontend/` - пользовательский интерфейс.
- `content/` - markdown-файлы уроков и JSON-файлы тестов.
- `infra/` - Docker Compose и переменные окружения для запуска в Docker.

## 4. Роли пользователей

### ADMIN

Доступ к административной панели и защищенным учебным API.

Возможности:

- вход в систему;
- просмотр списка пользователей;
- создание пользователя;
- редактирование пользователя;
- архивирование пользователя;
- просмотр курса и уроков;
- скачивание сертификата при выполнении условий завершения курса.

### STUDENT

Доступ к прохождению курса.

Возможности:

- вход в систему;
- просмотр личного кабинета;
- просмотр курса и прогресса;
- прохождение доступных уроков;
- отправка ответов на тест;
- скачивание сертификата после завершения курса.

## 5. Авторизация и безопасность

Авторизация выполняется по логину и паролю через endpoint:

```text
POST /api/auth/login
```

После успешного входа backend возвращает JWT-токен. Frontend использует токен для защищенных запросов.

Требования:

- пароль хранится в базе в виде bcrypt-хеша;
- архивный пользователь не может войти в систему;
- защищенные endpoint'ы требуют Bearer JWT;
- доступ к административным endpoint'ам разрешен только роли `ADMIN`;
- роли пользователей: `ADMIN`, `STUDENT`;
- статусы пользователей: `ACTIVE`, `ARCHIVED`.

Стартовый администратор создается миграцией и дополнительно синхронизируется при запуске backend:

```text
login: admin
password: secret
role: ADMIN
status: ACTIVE
```

## 6. Пользователь

Поля пользователя:

- `id` - UUID;
- `name` - имя пользователя;
- `login` - уникальный логин;
- `password_hash` - хеш пароля;
- `role` - `ADMIN` или `STUDENT`;
- `status` - `ACTIVE` или `ARCHIVED`;
- `created_at` - дата создания;
- `updated_at` - дата обновления.

Администратор может:

- получить список пользователей;
- получить пользователя по UUID;
- создать пользователя;
- обновить имя, логин, роль, статус и при необходимости пароль;
- архивировать пользователя без физического удаления из базы.

## 7. Курс и уроки

Система поддерживает один основной курс.

Курс содержит последовательный набор уроков. Порядок уроков определяется полем `position`.

Каждый урок содержит:

- UUID;
- порядковый номер;
- название;
- markdown-контент;
- тест;
- правило прохождения теста.

Правила доступности:

- первый урок доступен сразу;
- следующий урок открывается только после успешного прохождения предыдущего;
- заблокированный урок нельзя открыть через прямой API-запрос;
- пройденные уроки можно открывать и проходить повторно.

Статусы урока для пользователя:

- `LOCKED` - урок заблокирован;
- `AVAILABLE` - урок доступен;
- `PASSED` - урок успешно пройден.

## 8. Контент курса

Контент хранится в файловой системе:

- markdown-уроки: `content/lessons`;
- JSON-тесты: `content/tests`.

При запуске backend выполняет импорт и синхронизацию контента с базой данных.

Требования:

- редактирование курса через интерфейс не предусмотрено;
- изменение учебного контента выполняется через файлы в `content`;
- после изменения файлов backend должен быть перезапущен для повторной синхронизации.

## 9. Тестирование уроков

Тест каждого урока описывается JSON-файлом.

Поддерживаются вопросы с одним правильным вариантом ответа.

Для каждого урока задается собственный проходной процент.

При выдаче урока:

- backend возвращает markdown-контент;
- backend возвращает вопросы и варианты ответов;
- признак правильного ответа клиенту не передается;
- варианты ответов перемешиваются при каждой выдаче.

При отправке результата:

- клиент должен передать ровно один ответ на каждый вопрос урока;
- backend проверяет, что вопросы относятся к текущему уроку;
- backend проверяет, что ответы относятся к соответствующим вопросам;
- backend считает процент правильных ответов;
- если процент не ниже проходного, урок считается пройденным;
- если процент ниже проходного, прогресс урока не обновляется.

При успешном прохождении:

- создается или обновляется запись прогресса;
- фиксируется `completed_at` при первом успешном прохождении;
- сохраняется лучший результат `best_score_percent`;
- открывается следующий урок, если он существует;
- после последнего урока курс считается завершенным.

## 10. Прогресс

Прогресс хранится отдельно для каждого пользователя и урока.

Поля прогресса:

- `id` - UUID;
- `user_id` - UUID пользователя;
- `lesson_id` - UUID урока;
- `best_score_percent` - лучший результат по уроку;
- `passed` - признак прохождения;
- `completed_at` - дата первого успешного прохождения;
- `created_at`;
- `updated_at`.

Для курса рассчитываются:

- количество пройденных уроков;
- общее количество уроков;
- процент прохождения;
- признак полного завершения курса.

## 11. Личный кабинет студента

Личный кабинет показывает:

- данные текущего пользователя;
- основной курс;
- процент прохождения;
- список уроков;
- статус каждого урока;
- лучший результат по пройденным урокам.

Студент может перейти только к доступным или уже пройденным урокам.

## 12. Административная панель

Административная панель предназначена для управления пользователями.

Функции:

- просмотр всех пользователей;
- создание пользователя с ролью `ADMIN` или `STUDENT`;
- редактирование пользователя;
- изменение статуса пользователя;
- архивирование пользователя.

Регистрация пользователей через публичную форму в MVP отсутствует.

## 13. Завершение курса и сертификат

Курс считается завершенным, если пользователь успешно прошел все уроки основного курса.

После завершения курса становится доступно скачивание сертификата:

```text
GET /api/certificate
```

Текущая реализация сертификата:

- формат: PDF;
- генерация на backend через ReportLab;
- доступна только после полного прохождения курса;
- PDF содержит имя пользователя;
- скачивание доступно повторно.

Если пользователь не завершил курс, backend возвращает ошибку `COURSE_NOT_COMPLETED`.

## 14. API

Основная документация API доступна после запуска backend:

```text
Swagger: http://localhost:8080/docs
ReDoc:   http://localhost:8080/redoc
OpenAPI: http://localhost:8080/openapi.json
```

Основные endpoint'ы:

| Метод | Путь | Назначение | Доступ |
| --- | --- | --- | --- |
| `POST` | `/api/auth/login` | Авторизация | публичный |
| `GET` | `/api/admin/users` | Список пользователей | `ADMIN` |
| `GET` | `/api/admin/users/{id}` | Пользователь по UUID | `ADMIN` |
| `POST` | `/api/admin/users` | Создание пользователя | `ADMIN` |
| `PUT` | `/api/admin/users/{id}` | Обновление пользователя | `ADMIN` |
| `DELETE` | `/api/admin/users/{id}` | Архивирование пользователя | `ADMIN` |
| `GET` | `/api/course` | Информация о курсе | `ADMIN`, `STUDENT` |
| `GET` | `/api/course/lessons` | Список уроков | `ADMIN`, `STUDENT` |
| `GET` | `/api/course/lessons/{lesson_id}` | Содержимое урока и тест | `ADMIN`, `STUDENT` |
| `POST` | `/api/course/lessons/{lesson_id}/submit` | Отправка ответов | `ADMIN`, `STUDENT` |
| `GET` | `/api/course/progress` | Прогресс курса | `ADMIN`, `STUDENT` |
| `GET` | `/api/certificate` | PDF-сертификат | `ADMIN`, `STUDENT` |
| `GET` | `/api/health` | Проверка доступности backend | публичный |

## 15. База данных

Используется PostgreSQL.

Схема создается Alembic-миграциями.

Основные таблицы:

- `users`;
- `courses`;
- `lessons`;
- `test_questions`;
- `test_answers`;
- `lesson_pass_rules`;
- `lesson_progress`.

Все основные сущности используют UUID.

Для PostgreSQL включается расширение `pgcrypto`, чтобы использовать `gen_random_uuid()`.

## 16. Запуск

Подробная инструкция по запуску находится в [руководстве локального запуска](./Руководство%20локального%20запуска.md).

Основные сценарии:

- демонстрация: весь проект в Docker Compose;
- разработка: PostgreSQL в Docker, backend и frontend локально.

## 17. Ограничения MVP

- поддерживается один основной курс;
- один язык интерфейса - русский;
- публичная регистрация пользователей отсутствует;
- восстановление пароля отсутствует;
- редактирование курса через интерфейс отсутствует;
- аналитика и расширенная статистика отсутствуют;
- журнал пользовательских действий отсутствует;
- email-уведомления отсутствуют;
- автоматические тесты отсутствуют;
- сертификат имеет минимальное PDF-оформление;
- загрузка пользовательских файлов отсутствует.

- добавить редактирование курса через административный интерфейс;
- добавить автоматические тесты backend и frontend;
- добавить экспорт пользователей и прогресса;
